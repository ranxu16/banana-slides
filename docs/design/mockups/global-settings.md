# 全局配置中心 V2 设计稿

## 高保真稿

- 静态 UI 文件：`docs/design/mockups/global-settings-v2.html`
- 用途：作为 `frontend/src/pages/Settings.tsx` 重构前的视觉验收稿，先确认整体样式、信息层级、左侧导航、配置画布、字段行、模型能力卡、服务测试和 sticky 保存条。
- 当前版本：桌面优先，同时包含移动端断点约束；仅用于设计确认，不连接真实接口。
- 本轮补充：全局配置中心不是只有管理员全局配置，还要支持每个用户配置自己的模型、密钥或账号连接，并能真正参与生成流水线的生效值计算。
- 最新调整：全局配置一级入口下先分成“全局默认配置 / 个人配置”两个二级入口；能力接入方式降级为说明模块，不作为主配置 section。
- 操作优先级：主界面优先展示可编辑控件、当前生效值、测试按钮、保存按钮和回退按钮；解释说明只能作为短帮助、帮助抽屉或低权重提示出现。

## 设计目标

全局配置中心要从“所有表单堆在一页”改成“配置工作台”。用户进入页面后，先知道当前系统是否可用，再按任务进入对应配置组。页面不追求装饰感，重点是扫描、定位、保存、测试和排错效率。

## 视觉布局

### 1. 页面头部

页面头部由 AppShell 承接，不再在内容区重复大标题。

内容区第一行是操作条：

```text
┌──────────────────────────────────────────────────────────────┐
│ 全局默认策略                         已保存 12:31  ● 服务正常 │
│ OpenAI/ChatGPT 为推荐主线，项目可显式覆盖。        [测试] [保存]│
└──────────────────────────────────────────────────────────────┘
```

要求：

- 不使用大面积渐变、英雄区、营销式文案。
- 保存状态和服务状态放在同一条操作区，不再散落在 toast 和底部按钮。
- “测试连接”是主动按钮，不能自动触发。
- 第一屏文字说明要短，不能挤压二级入口、配置选择控件和保存/测试/回退按钮。

### 2. 健康状态卡

只保留 3 张横向状态卡，减少当前页面顶部信息噪音。

```text
┌──────────────┬──────────────┬──────────────┐
│ OpenAI 主线  │ 模型配置      │ 敏感密钥      │
│ 已配置/待配置│ 5/6 已完成    │ 4 项已配置    │
└──────────────┴──────────────┴──────────────┘
```

每张卡只展示一个判断，不放长说明。健康状态卡只做状态提示，不承担配置说明职责。

### 3. 左侧导航

左侧导航固定宽度 240px，粘在顶部。每个导航项必须与右侧一个 section 一一对应。

```text
搜索配置...

基础配置
AI 模型
生成与导出
文件解析
服务连接
兼容高级
覆盖链路
服务测试
```

规则：

- 一个导航项只能跳到一个右侧 section。
- 右侧不能出现左侧没有入口的大块配置。
- 当前激活项高亮，滚动时同步。
- 不在导航项下面塞长描述；描述放右侧 section header。

### 4. 二级入口

全局配置这个一级入口下必须先分成两个二级入口：

```text
全局默认配置 | 个人配置
```

规则：

- 全局默认配置：管理员维护组织默认模型、API 执行凭据、企业代理、生成导出策略、解析服务和服务测试。
- 个人配置：当前用户维护自己的模型、订阅/账号连接、个人 API Key、能力偏好和是否回退到全局默认。
- 二级入口放在页面内容顶部，优先级高于左侧分组导航。
- 左侧导航只显示当前二级入口下的配置分组，不同时混放全局和个人配置。
- 能力接入方式不是主配置入口，只作为说明或帮助模块出现。
- 个人配置页的主区域必须是可操作表单；说明内容只能附在字段下方一行或进入“查看详情”。

## 配置生效模型

全局配置中心要支持管理员默认配置，也要支持每个用户配置自己的模型。模型、密钥和账号连接不能只停留在页面展示，需要进入后端配置解析和生成流水线。

最终生效值按下面链路计算：

```text
系统默认 → 全局配置 → 用户个人配置 → 项目覆盖 → 当前生效值
```

规则：

- 用户个人配置默认继承全局配置，不强迫每个用户填写 API Key。
- 用户可以按能力单独覆盖模型，例如只覆盖文本生成模型，不影响图片生成和可编辑 PPTX 拆层。
- 项目覆盖优先级高于用户个人配置，用于具体项目稳定复现。
- 所有模型字段必须展示来源：系统默认 / 全局配置 / 用户个人配置 / 项目覆盖。
- 管理员可以配置全局默认、组织级 API、企业代理和账号连接策略，但不能查看用户个人密钥明文。
- 普通用户只能配置自己的模型、密钥和账号连接，不能修改全局默认。
- 生成、导出、图片描述、可编辑 PPTX 拆层、gpt-image-2 独立元素生成都必须通过统一的配置 resolver 获取当前生效值，不能直接读取某一个全局配置字段。

## 凭据与账号连接模型

模型配置的凭据来源不只一种，后端和 UI 都要明确区分：

```text
API Key
API 组织 / 项目计费
企业级后端代理
用户个人 API Key
用户账号 / 订阅连接
```

规则：

- API Key、API 组织计费和企业代理可以作为自动生成流水线的执行凭据。
- 用户个人 API Key 可以作为当前用户的执行凭据，但必须加密存储、脱敏展示、按用户隔离。
- ChatGPT / OpenAI 账号或订阅连接要作为真实的账号能力接入，而不只是 UI 展示；它可以用于账号状态、工作区身份、人工辅助入口、Codex/ChatGPT 登录态或企业授权能力。
- 订阅连接不能和 API 平台计费混为一个字段；页面必须分别展示“账号/订阅状态”和“API 执行凭据状态”。
- 如果某类订阅或账号连接不能用于后台自动化 API 调用，后端 resolver 必须返回明确原因，前端在服务测试里展示，而不是静默失败。

## 能力接入方式说明

配置界面需要明确告诉用户：哪些能力适合用 ChatGPT/Codex/账号订阅连接，哪些能力建议或必须用 API。这个信息不作为主界面里的大配置区，而是放在服务连接、个人配置或帮助抽屉里的说明模块。

```text
能力                         推荐接入方式              订阅/账号连接作用
一问生成 PPT 大纲             订阅/账号连接 或 API        可作为默认创作入口；管理员可强制 API
生成页面描述                  订阅/账号连接 或 API        可作为默认创作入口；失败时可切回全局默认
自然语言修改页面              订阅/账号连接 或 API        适合个人订阅/账号连接；自动化批量任务可走 API
图片描述 / 视觉理解           API / 企业代理             订阅连接仅展示账号能力，后台识别建议走 API
图片生成                     API / 企业代理             后台生成图片走 API
可编辑 PPTX 视觉拆层          API / 企业代理             必须可程序化调用，不走 ChatGPT 网页订阅
gpt-image-2 独立元素生成      API / 企业代理             必须可程序化调用，不走 ChatGPT 网页订阅
导出任务队列                  API / 企业代理             必须服务端可追踪、可重试、可审计
人工辅助 / 打开 ChatGPT       订阅/账号连接              可用订阅能力
Codex 辅助操作                Codex/ChatGPT 登录态       可用账号连接，取决于工作区授权
```

界面表现：

- 不在主配置流里放大表格。
- 在“个人配置”的账号连接区域展示简短说明和“查看能力接入说明”入口。
- 说明弹层或帮助抽屉中展示：能力、推荐方式、订阅是否可用、API 是否就绪、不可用原因。
- 主页面中说明文案不能超过可操作控件的视觉权重；不能出现大段说明卡抢占首屏。
- 对后台自动化能力标记“建议 API”或“必须 API”，对人工辅助能力标记“可用订阅”。
- 如果用户只连接了 ChatGPT 订阅但没有 API 执行凭据，后台生成按钮旁要提示“订阅已连接，但该能力需要 API / 企业代理”。
- 如果企业代理已配置，可作为 API Key 的组织级替代方式，显示为“企业代理可执行”。
- 这不是静态说明，后端要返回每项能力的 `execution_mode`、`subscription_supported`、`api_required`、`ready` 和 `reason`。

默认与回退：

- 默认状态下使用全局默认配置。
- 如果用户配置了个人配置，当前用户默认优先使用个人配置。
- 个人配置页必须提供总开关“强制使用全局默认”。
- 每个能力也必须能选择“继承全局默认”，用于个人配置局部失效时快速回退。
- 回退不能只改前端展示，后端 resolver 必须按 `force_global_default` 和能力级 `use_global_default` 计算最终生效值。

### 5. 右侧配置画布

## 分组设计

### 基础配置

用途：系统默认 provider、API Base、全局 API Key。

字段：

- 默认 Provider
- API Base URL
- API Key 状态
- 配置来源说明

重点：

- OpenAI/ChatGPT 是推荐主线，放在首位。
- Gemini、LazyLLM、其他兼容 provider 放入“兼容高级”或二级选项，不与 OpenAI 平级抢主视觉。

### AI 模型

用途：把模型按能力拆开，而不是把模型名混在一组输入框里。

子块：

- 文本生成模型
- 视觉理解模型
- 图片生成模型
- 图片描述模型
- 可编辑 PPTX 结构分析模型
- 可编辑 PPTX 独立元素生成模型

关键默认策略：

- 文本生成：OpenAI/ChatGPT 主线。
- 视觉理解：OpenAI 视觉模型主线。
- 图片生成：`gpt-image-2` 主线。
- 图片描述：OpenAI 视觉模型主线，可兼容其他 provider。
- 可编辑 PPTX 结构分析：GPT 视觉拆层。
- 可编辑 PPTX 独立元素生成：`gpt-image-2`。

每个子块必须展示：

- 当前值
- 来源：系统默认 / 全局配置 / 用户个人配置 / 项目覆盖
- Provider
- API Key 状态
- 测试按钮
- 字段级错误

### 个人配置

用途：允许每个用户使用自己的模型、API Key、企业代理或账号连接，也可以随时切回全局默认。

展示：

```text
个人配置总策略：自动使用个人配置 / 强制使用全局默认
个人模型策略：继承全局默认 / 使用我的订阅账号 / 使用我的 API 配置 / 使用企业代理
能力覆盖：文本生成、视觉理解、图片生成、图片描述、可编辑 PPTX 结构分析、独立元素生成
凭据状态：未配置 / 已配置，长度 N / 使用组织代理 / 账号已连接但不可用于 API 执行
生效预览：当前用户在新建项目中的实际模型链路
```

重点：

- 功能层面默认使用默认配置；如果用户配置了个人配置，默认优先使用个人配置。
- 用户需要有一个总开关“强制使用全局默认”，用于个人配置失效时快速切回可用配置。
- 每个能力可以独立选择“继承全局默认”或“个人覆盖”。
- 大纲生成、页面描述生成、自然语言改写允许选择订阅/账号连接作为个人配置来源。
- 生成 PPTX、可编辑 PPTX 视觉拆层、gpt-image-2 独立元素生成、导出任务队列必须使用 API / 企业代理。
- 保存个人配置只影响当前用户，不影响其他用户和全局默认。
- 个人 API Key 仍按敏感字段处理，只显示状态和长度。
- 如果项目设置了覆盖，以项目覆盖为准，并在字段旁显示“已被项目覆盖”。
- 个人模型配置要有后端 API、权限隔离和 resolver 支持，不能只做前端本地状态。

### 生成与导出

用途：所有影响内容生成和导出表现的策略放一起。

子块：

- 输出语言
- 图片清晰度
- 描述生成模式
- 并发数
- PPTX/PDF 导出策略
- 可编辑 PPTX 错误处理策略
- 是否允许半成品

重点：

- “允许半成品”必须明确风险，默认不鼓励作为主线。
- 可编辑 PPTX 配置要链接到导出任务中心，方便排错。

### 文件解析

用途：参考文件和可编辑导出依赖服务。

子块：

- MinerU API Base
- MinerU Token
- 百度 OCR / 图像修复
- 解析失败处理

重点：

- 密钥只显示状态和长度。
- 解析服务测试放在本 section 内，也可以在服务测试汇总里重复出现。

### 服务连接

用途：第三方账号连接和服务状态。

子块：

- OpenAI OAuth
- ChatGPT / OpenAI 账号订阅连接状态
- API 组织与计费状态
- 企业代理 / 组织统一凭据
- ElevenLabs
- Codex/OpenAI 账号状态提示

重点：

- OAuth 连接状态要清楚。
- 连接失败必须给可操作的手动 callback 输入入口。
- 账号/订阅连接要是真实接入能力，不只是状态展示；后端需要保存连接状态、授权来源、可用范围和不可用于自动化执行时的原因。
- API Key、API 组织计费、企业代理、个人 API Key 和账号/订阅连接要分开建模。
- ChatGPT / OpenAI 订阅状态不能默认等同于 API 平台可执行额度；如果后台生成不能使用该连接，服务测试必须说明“账号已连接，但当前凭据不能用于自动化 API 调用”。

### 兼容高级

用途：低频、兼容、成本较高或风险较高的配置。

内容：

- LazyLLM 厂商 API Key
- Gemini 兼容配置
- OpenAI 图片 API 协议覆盖
- 文本推理预算
- 图片推理预算

重点：

- 默认折叠，但左侧导航能直接定位并展开。
- 不把兼容项放到第一屏抢主线。

### 覆盖链路

用途：说明系统默认、全局配置、用户个人配置和项目级配置如何共同决定当前生效值。

展示：

```text
系统默认 → 全局配置 → 用户个人配置 → 项目覆盖 → 当前生效值
```

字段：

- 用户个人模型覆盖
- 用户个人 API Key / 企业代理 / 账号连接选择
- 项目导出方法覆盖
- 项目画面比例覆盖
- 项目可编辑 PPTX 视觉分析开关
- 项目是否允许半成品

当前如果后端没有完整来源字段，先展示说明态，不伪造来源。

### 服务测试

用途：集中测试服务，不自动触发。

测试项：

- 文本模型
- 视觉理解模型
- 图片生成模型
- 图片描述模型
- MinerU PDF 解析
- 百度 OCR
- 百度图像修复
- ElevenLabs

每项测试结果必须展示：

- 状态：未测试 / 测试中 / 成功 / 失败
- 失败 message
- 最近测试时间
- 使用的配置来源

## 组件规范

### 操作优先级

主界面的视觉优先级从高到低：

```text
当前生效方式 / 下拉选择 / 按钮
保存、测试、回退、继承全局
状态与来源
短帮助文本
详细说明入口
```

规则：

- 能力卡首先展示当前可选值，而不是解释文字。
- 每个关键能力必须有“测试”或“继承全局/个人 API”等明确动作。
- “强制使用全局默认”必须是显眼按钮，不藏在说明里。
- 说明内容不超过一行；长说明进入帮助抽屉。
- 第一屏可操作控件数量要多于解释说明块数量。

### FormRow

```text
Label              Control                         SourceBadge
Description        InlineError / HelpText
```

规则：

- Label 宽度统一，Control 对齐。
- 错误显示在字段下方，不只 toast。
- SourceBadge 放在右侧，显示来源。
- 密钥字段使用“已配置，长度 N”状态和“重新填写/清空”动作。

### SectionHeader

```text
Title
Description
Status summary chips
```

规则：

- 每个 section 顶部只解释本组职责。
- 不在每个卡片里重复 OpenAI 推荐文案。

### StickySaveBar

底部保存条只在有未保存修改时出现。

```text
3 项未保存                 [放弃更改] [保存更改]
```

无修改时不固定占位。

## 交互状态

- Loading：骨架屏，不显示空白大卡。
- Load failed：顶部错误条 + 重试按钮。
- Dirty：字段行左侧出现细色条，底部显示保存条。
- Save success：更新保存时间，不只 toast。
- Save failed：保留 dirty 状态，字段错误优先展示。
- Test running：测试按钮 loading，其他测试项不受影响。
- Test failed：错误 message 原样展示。

## 移动端

移动端不展示固定左侧导航，改为顶部分组 select。

```text
[AI 模型 v]
当前 section 内容
底部保存条
```

按钮不挤压正文，低频动作进入更多菜单。

## 不做的事

- 不做 hero。
- 不做渐变背景。
- 不做大圆角装饰卡。
- 不在一屏展示全部配置字段。
- 不把兼容 provider 和 OpenAI 主线放成同等视觉权重。
- 不自动测试服务。

## 验收

- 左侧导航每项都能定位到唯一右侧 section。
- 第一屏能看出系统是否配置完整。
- AI 模型区能明确看到 `gpt-image-2` 独立元素生成能力。
- 密钥不明文回显。
- 字段级错误位置明确。
- 未保存修改有明显提示。
- 服务测试失败原因不用打开开发者工具即可读。
